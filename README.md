# Vibe

AI-powered application builder. Describe a web app in plain language; receive a live, running Next.js application — generated by an LLM agent, executed in an isolated cloud sandbox, and streamed back as a preview URL — without touching a terminal.

---

## System design

```
Browser ──tRPC──► Next.js 15 ──Prisma──► PostgreSQL
                      │
                  Inngest event
                      │
               ┌──────▼──────────────────────┐
               │  Inngest worker              │
               │  @inngest/agent-kit          │
               │                             │
               │  GPT-4.1 ──codegen──► files │
               └──────────────┬──────────────┘
                              │ E2B SDK
                   ┌──────────▼────────────┐
                   │  E2B Cloud Sandbox     │
                   │  Node 21 + Next.js 15  │
                   │  Shadcn pre-installed  │
                   │  Turbopack dev server  │
                   └──────────┬────────────┘
                              │ public HTTPS URL
                   Fragment ──┘ stored in DB
                   Browser polls → renders iframe
```

The critical design decision: AI code execution runs **entirely off the Next.js request thread** via Inngest. The HTTP response returns immediately after enqueuing the job. The browser polls React Query until a `Fragment` row appears. This keeps p99 request latency low regardless of LLM or sandbox cold-start time.

## Stack

| | |
|---|---|
| **Framework** | Next.js 15 · App Router · standalone output |
| **Language** | TypeScript 5 · end-to-end type safety via tRPC |
| **API** | tRPC v11 + TanStack Query v5 — no REST, no codegen |
| **Auth** | Clerk — managed sessions, OAuth, plan-based access control |
| **Database** | PostgreSQL 16 · Prisma 6 (custom client output at `src/generated/prisma`) |
| **Jobs** | Inngest + `@inngest/agent-kit` — durable, retryable background functions |
| **AI** | `gpt-4.1` for code generation · `gpt-4o` for titles and summaries |
| **Sandboxes** | E2B Cloud — isolated ephemeral runtimes; pre-built template cuts cold-start from ~60 s to ~5 s |
| **UI** | Tailwind v4 · Shadcn/ui · Radix primitives · `react-resizable-panels` |

## Quickstart (Docker)

### 1. Configure

```bash
git clone <repo> && cd vibe
cp env.example .env
# Fill in every value — see Configuration below
```

### 2. Build the E2B sandbox template

> One-time step. The AI agent cannot create sandboxes without a published template.

```bash
e2b auth login
cd sandbox-templates/nextjs
e2b template build   # prints a template ID — copy it into the agent code
cd ../..
```

### 3. Run

```bash
docker compose up --build                    # postgres + migrate + app
docker compose --profile dev up --build      # + Inngest dashboard at :8288
```

Migrations run automatically before the app starts. Open `http://localhost:3000`.

## Local development

```bash
# Database (or bring your own PostgreSQL 16)
docker run -d --name vibe-pg \
  -e POSTGRES_DB=vibe -e POSTGRES_USER=vibe -e POSTGRES_PASSWORD=secret \
  -p 5432:5432 postgres:16-alpine

npm install
npx prisma migrate dev

# Terminal 1 — background job processor (required for AI responses)
npx inngest-cli@latest dev

# Terminal 2 — app
npm run dev
```

Without the Inngest dev server running, prompts will submit but no AI response will ever arrive.

## Configuration

All variables must be present before `next build` or container start. `NEXT_PUBLIC_*` values are **baked into the JS bundle at build time** — changing them requires a rebuild. `docker-compose.yml` forwards them automatically from `.env` as build args.

### Core

| Variable | Required | Description |
|---|---|---|
| `NEXT_PUBLIC_APP_URL` | ✓ | Public origin, no trailing slash. e.g. `https://app.example.com` |
| `DATABASE_URL` | ✓ | PostgreSQL connection string. In Compose, this is overridden to target the internal `postgres` service. |

### OpenAI

| Variable | Required | Description |
|---|---|---|
| `OPENAI_API_KEY` | ✓ | Must have access to `gpt-4.1` and `gpt-4o`. |

### E2B

| Variable | Required | Description |
|---|---|---|
| `E2B_API_KEY` | ✓ | Authenticates both template pushes and runtime sandbox creation. |

### Clerk

| Variable | Required | Description |
|---|---|---|
| `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` | ✓ | Build-time. From Clerk dashboard → API Keys. |
| `CLERK_SECRET_KEY` | ✓ | Server-side only. Never exposed to the client. |
| `NEXT_PUBLIC_CLERK_SIGN_IN_URL` | — | Defaults to `/sign-in` |
| `NEXT_PUBLIC_CLERK_SIGN_UP_URL` | — | Defaults to `/sign-up` |
| `NEXT_PUBLIC_CLERK_SIGN_IN_FALLBACK_REDIRECT_URL` | — | Defaults to `/` |
| `NEXT_PUBLIC_CLERK_SIGN_UP_FALLBACK_REDIRECT_URL` | — | Defaults to `/` |

### Inngest (production only)

| Variable | Required | Description |
|---|---|---|
| `INNGEST_EVENT_KEY` | Prod | From Inngest dashboard → your app. |
| `INNGEST_SIGNING_KEY` | Prod | Verifies that incoming webhook requests originate from Inngest. |

### Docker Compose database bootstrap

| Variable | Default | Description |
|---|---|---|
| `POSTGRES_DB` | `vibe` | Database name for the `postgres` service. |
| `POSTGRES_USER` | `vibe` | Database user. |
| `POSTGRES_PASSWORD` | — | Required. Set in `.env`. |

## Database

Schema: `Project → Message[] → Fragment?` · `Usage` (credit ledger)

```bash
npx prisma migrate dev --name <description>   # development
npx prisma migrate deploy                     # production / CI
npx prisma studio                             # browser-based inspector
npx prisma migrate reset                      # nuke + replay (dev only)
```

Prisma's generated client lives at `src/generated/prisma` (non-default). The `postinstall` script runs `prisma generate` automatically after `npm install`.

## Deployment

### Self-hosted

```bash
docker build \
  --build-arg NEXT_PUBLIC_APP_URL=https://app.example.com \
  --build-arg NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY=pk_live_... \
  -t vibe:latest .

docker run -d -p 3000:3000 \
  --env-file .env.production \
  vibe:latest
```

Use a managed PostgreSQL service (RDS, Neon, Supabase). Append `?sslmode=require` to `DATABASE_URL`.

### Vercel

Import the repo. Set all env vars in the Vercel dashboard. Register `https://your-domain.com/api/inngest` as the webhook in your Inngest app settings.

### Production checklist

- [ ] `NEXT_PUBLIC_APP_URL` is HTTPS with no trailing slash
- [ ] Clerk redirect URLs include the production domain
- [ ] `DATABASE_URL` includes `?sslmode=require`
- [ ] `prisma migrate deploy` has run against the production DB
- [ ] E2B sandbox template is built and template ID is wired into the agent
- [ ] Inngest webhook registered at `https://your-domain/api/inngest`

## Troubleshooting

**No AI response after submitting a prompt**
Inngest dev server is not running. Start it with `npx inngest-cli@latest dev` or include `--profile dev` in Docker Compose. Inspect failed runs at `http://localhost:8288`.

**`PrismaClientInitializationError: Unable to require(...libquery_engine-linux-musl...)`**
Alpine Linux binary is missing. Verify `prisma/schema.prisma` has `binaryTargets = ["native", "linux-musl-openssl-3.0.x"]` and the Dockerfile copies `src/generated` into the runner stage. Rebuild with `--no-cache`.

**E2B sandbox creation fails**
`E2B_API_KEY` is missing or the sandbox template has not been built. Run `e2b template build` from `sandbox-templates/nextjs/` and verify the template ID matches what the agent references.

**Clerk error in browser after Docker build**
`NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` is embedded at build time. A stale cached layer will serve the wrong value. Run `docker compose build --no-cache app`.
